# Meteor Object Pooling Guide

## 목적

현재 운석은 `Instantiate` / `Destroy` 방식으로 생성과 제거를 반복한다.  
오브젝트 풀링으로 바꾸면 생성 비용과 GC 부담을 줄일 수 있지만, 시작 시점에 어느 정도의 오브젝트를 미리 확보할지 결정해야 한다.

이 문서는 아래 3가지를 정리한다.

1. 풀 크기를 어떤 방식으로 산정할지
2. 풀링 적용 전후에 무엇이 달라지는지
3. 실제 게임에서 차이가 있는지 빠르게 프로파일링하는 방법

---

## 1. 풀 크기 산정 방식

### 핵심 원칙

풀 크기는 "동시에 살아 있을 수 있는 운석 수"를 기준으로 잡아야 한다.  
즉, 단순히 "많아 보이니까 30개"처럼 정하는 것이 아니라 아래 순서로 계산한다.

1. 최소 스폰 간격 확인
2. 운석이 살아 있을 수 있는 최대 시간 확인
3. 이론상 동시에 존재 가능한 최대 수 계산
4. 실제 플레이에서 관측한 피크 값으로 보정
5. 여유 버퍼를 추가해 최종 풀 크기 결정

### 현재 프로젝트 기준으로 봐야 하는 값

`MeteorSpawner.cs`

- `minimumSpawnInterval = 0.07`
- `startingSpawnInterval = 0.12`
- `useVisibleMeteorCap = false`
- `startingMaxVisibleMeteors = 14`
- `maximumVisibleMeteors = 28`

`Meteor.cs`

- `maxLifetime = 12`

### 1차 계산: 이론상 최악의 동시 존재 수

가장 단순하고 보수적인 계산은 아래 공식이다.

```text
theoretical_max_concurrency = ceil(maxLifetime / minimumSpawnInterval)
```

현재 값에 대입하면:

```text
ceil(12 / 0.07) = ceil(171.42...) = 172
```

이 숫자는 "매우 보수적인 최악의 경우"다.  
즉, 모든 운석이 빠르게 정리되지 않고 거의 최대 수명까지 남아 있다고 가정한 값이다.

실제 플레이에서는 많은 운석이 더 빨리 행성에 닿거나 화면 바깥으로 사실상 의미를 잃기 때문에 평균 동시 개체 수는 이보다 낮을 가능성이 크다.

### 2차 계산: 디자인 제한이 있으면 그 값을 우선 사용

`useVisibleMeteorCap`를 켜면 풀 크기는 훨씬 쉽게 결정할 수 있다.

#### `useVisibleMeteorCap`의 의미

이 옵션은 "현재 화면이나 라운드에서 동시에 활성 상태로 존재할 수 있는 운석 수"를 제한하는 스폰 규칙이다.

현재 코드 기준으로는 [MeteorSpawner.cs](c:/MeteoRun/Assets/Scripts/MeteorSpawner.cs#L70)에서 `activeMeteors.Count >= GetCurrentVisibleMeteorCap()`가 되면 새 운석을 스폰하지 않는다.

즉, 이 값은 아래처럼 이해하면 된다.

- `true`: 활성 운석 수 상한을 두고 그 이상은 스폰하지 않음
- `false`: 활성 운석 수 상한 없이 스폰 규칙과 수명만으로 개체 수가 결정됨

중요한 점은 `useVisibleMeteorCap`이 "풀 크기"를 직접 정하는 옵션은 아니라는 것이다.

- visible cap: 동시에 꺼내 쓸 수 있는 활성 운석 수 제한
- pool size: 메모리에 미리 준비해 둘 총 운석 오브젝트 수

오브젝트 풀을 도입하면 보통 `pool size >= visible cap`이 되도록 잡는다.

추천 공식:

```text
pool_size = maximumVisibleMeteors + safety_buffer
```

버퍼는 보통 `4 ~ 8`이면 충분하다.

예시:

```text
28 + 4 = 32
28 + 8 = 36
```

즉, visible cap을 켠다면 현재 프로젝트에서는 `32 ~ 36` 정도가 첫 고정 풀 크기로 적절하다.

### visible cap을 끄고 갈 경우의 권장 방식

지금처럼 `useVisibleMeteorCap = false`면 처음부터 고정 풀 크기를 하나로 박는 것보다 아래 방식이 더 안전하다.

#### 권장 전략: 하이브리드 방식

1. 초기 풀을 너무 작지 않게 미리 생성
2. 부족하면 런타임 중 일정 단위로 확장
3. 테스트에서 측정한 피크 사용량을 기준으로 최종 고정 크기 확정

현재 프로젝트에서는 아래 값을 권장한다.

- 초기 풀 크기: `64`
- 확장 단위: `16`
- 최종 배포용 고정 풀 크기: 프로파일링 후 결정

이 방식을 추천하는 이유:

- 처음부터 `172 ~ 192`개를 모두 미리 만들면 메모리 예약량이 커진다.
- 반대로 `20 ~ 30`개로 시작하면 런타임 확장이 너무 자주 발생할 수 있다.
- `64`는 과하게 무겁지 않으면서 초반 플레이를 대부분 커버할 가능성이 높다.

### 풀이 부족하면 런타임에 어떻게 할 것인가

오브젝트 풀에서는 스폰 시점에 "비활성 상태로 대기 중인 운석"이 하나도 없을 수 있다. 이때의 런타임 정책을 미리 정해야 한다.

대표적인 선택지는 3가지다.

#### 1. 즉시 확장

- 동작: 풀이 비면 새 오브젝트를 추가 생성하고 풀 크기를 늘린다.
- 장점: 게임플레이가 끊기지 않고 스폰 손실이 없다.
- 단점: 확장 시점에 작은 CPU/메모리 스파이크가 생길 수 있다.

#### 2. 스폰 생략

- 동작: 풀이 비면 그 프레임의 스폰을 포기한다.
- 장점: 런타임 추가 할당이 없어 가장 안정적이다.
- 단점: 의도한 난이도보다 운석이 적게 나와 게임 템포가 바뀔 수 있다.

#### 3. 오래된 오브젝트 재사용

- 동작: 가장 오래된 활성 운석이나 화면 밖 운석을 강제로 회수해서 재사용한다.
- 장점: 풀 크기를 고정하기 쉽다.
- 단점: 시각적으로 어색하고 디버깅이 어려워질 수 있다.

이 프로젝트에서는 아래 방식이 가장 현실적이다.

- 기본 정책: `16`개 단위로 런타임 확장
- 안전장치: 확장이 일정 횟수 이상 반복되면 경고 로그 또는 디버그 HUD에 표시
- 최종 목표: 프로파일링 후 확장이 거의 발생하지 않는 크기로 고정

즉, 개발 중에는 "확장 가능"으로 두고 실제 사용량을 측정한 다음, 배포 직전에는 적절한 고정 크기로 확정하는 흐름이 좋다.

실무적으로는 아래 카운터를 같이 두는 것이 좋다.

- `Pool Miss Count`: 꺼내 쓸 비활성 운석이 없었던 횟수
- `Pool Expansion Count`: 풀을 실제로 확장한 횟수
- `Peak Active Meteors`: 테스트 중 동시에 활성 상태였던 최대 운석 수

이 값들이 높으면 초기 풀 크기가 작다는 뜻이다.

### 최종 배포용 풀 크기 결정 절차

실제 배포용 숫자는 아래 순서로 정하는 것이 가장 정확하다.

1. 스트레스 테스트 시나리오를 만든다.
2. 플레이 10회 이상에서 `동시에 활성 상태였던 운석 최대 개수`를 기록한다.
3. 그중 가장 큰 값에 `10% ~ 20%` 버퍼를 더한다.
4. 그 수치를 최종 고정 풀 크기로 사용한다.

추천 공식:

```text
final_pool_size = ceil(observed_peak_active * 1.15)
```

예시:

- 관측 피크가 `54`면 최종 풀 크기는 `63`, 실무적으로는 `64`
- 관측 피크가 `71`이면 최종 풀 크기는 `82`, 실무적으로는 `80` 또는 `96`

### 이 프로젝트에 대한 실전 권장안

#### 선택지 A. 안정성 우선, 런타임 확장 없이 가기

- visible cap 유지 안 함
- 풀 크기 `192` 전후로 고정

장점:

- 런타임 확장 거의 없음
- 테스트 중 풀 부족으로 인한 예외 상황이 적음

단점:

- 시작 시점 메모리 예약량이 큼
- 실제 평균 사용량이 낮다면 낭비가 큼

#### 선택지 B. 현재 프로젝트에 더 현실적인 권장안

- 초기 풀 `64`
- 부족 시 `16`개씩 확장
- 프로파일링 후 최종 고정 크기 확정

장점:

- 시작 메모리 부담이 낮음
- 실제 플레이 패턴에 맞춰 적정 크기를 찾기 쉬움

단점:

- 첫 확장 시점에 작은 비용이 발생할 수 있음

#### 선택지 C. visible cap을 켜는 경우

- `maximumVisibleMeteors = 28`
- 풀 크기 `32 ~ 36`

장점:

- 풀 크기 설계가 가장 쉬움
- 메모리 사용량 예측이 쉬움

단점:

- 난이도 설계가 cap에 의해 제한됨

## 2. 오브젝트 풀링 적용 전후 차이

### 적용 전: Instantiate / Destroy 방식

현재 방식의 특징:

- 스폰할 때마다 새 오브젝트를 생성한다.
- 정리할 때마다 `Destroy`를 호출한다.
- 생성/삭제가 반복되면서 메모리 할당과 해제가 자주 일어난다.

실제 영향:

- CPU 스파이크가 생길 수 있다.
- GC가 도는 프레임에서 끊김이 보일 수 있다.
- 운석 수가 많아질수록 프레임 타임이 흔들릴 가능성이 커진다.
- 에디터 Hierarchy에 `Meteor(Clone)`이 계속 생겼다가 사라진다.

### 적용 후: Object Pool 방식

풀링 방식의 특징:

- 시작 시점 또는 필요 시점에 운석 오브젝트를 미리 만들어 둔다.
- 스폰 시 새로 만들지 않고 비활성 오브젝트를 재사용한다.
- 제거 시 `Destroy` 대신 상태를 초기화하고 풀로 반환한다.

실제 영향:

- 런타임 `Instantiate` / `Destroy` 호출 횟수가 크게 줄어든다.
- `GC.Alloc`가 줄어들고 GC 스파이크도 줄어든다.
- 프레임 타임이 더 안정적이 된다.
- 생성/삭제 비용이 아니라 "재초기화 비용"만 남는다.

### 적용 전후를 한 줄로 비교하면

```text
Before: 자주 만들고 자주 버린다.
After : 한 번 만들어 두고 계속 재사용한다.
```

### 풀링을 써도 사라지지 않는 것

오브젝트 풀링은 "메모리를 0으로 만든다"는 뜻이 아니다.

오히려 특징은 아래와 같다.

- 시작 시점 메모리 예약량은 늘어날 수 있다.
- 대신 플레이 중 메모리 변동 폭과 GC 부담이 줄어든다.
- 즉, "총 사용량"보다 "변동성과 끊김"을 줄이는 데 효과가 크다.

### 풀링 적용 시 반드시 같이 챙겨야 하는 것

재사용할 때는 이전 상태가 남지 않도록 초기화해야 한다.

예시:

- 위치
- 회전
- 이동 방향
- 속도
- 수명 타이머
- 충돌 가능 여부
- `TrailRenderer` 상태
- `SpriteRenderer.enabled`
- `Rigidbody2D.simulated`
- 게임오버 충돌로 남겨 두었던 특수 상태

이 초기화가 빠지면 "풀링 버그"가 생긴다.  
즉, 오브젝트는 재사용되지만 이전 라운드의 흔적이 남아서 이상 동작을 일으킬 수 있다.

## 3. 빠르고 효율적인 프로파일링 방법

### 목표

풀링 적용 전후를 비교할 때는 "느낌"이 아니라 같은 조건에서 A/B 측정해야 한다.

가장 빠르고 눈에 띄게 확인하려면 아래 3개를 보면 된다.

1. `GC.Alloc`
2. `Instantiate` / `Destroy` 호출 비용
3. 프레임 타임 흔들림

### 가장 효율적인 측정 방법

#### 1. 비교 조건을 고정한다

둘을 비교할 때 조건이 다르면 결과가 의미가 없다.

고정할 항목:

- 같은 씬
- 같은 카메라 배율
- 같은 난이도 설정
- 같은 스폰 간격
- 같은 테스트 시간
- 가능하면 같은 랜덤 시드

권장 테스트 시나리오:

- `minimumSpawnInterval` 근처까지 빠르게 도달하거나 아예 고정
- `useVisibleMeteorCap`는 켜거나 끄더라도 양쪽 버전에서 동일하게 유지
- 테스트 길이 `60초`
- 플레이어 입력 없이 자동으로 계속 살아남는 상황 또는 재현 가능한 고정 입력 사용

가능하면 테스트 시작 전에 `Random.InitState(...)`로 랜덤을 고정하면 A/B 차이가 더 잘 보인다.

#### 2. 에디터보다 Development Build를 우선 사용한다

가장 빠른 확인은 에디터 Play Mode지만, 에디터 자체 오버헤드가 섞여서 결과가 흔들릴 수 있다.

권장 순서:

1. 에디터에서 빠르게 1차 확인
2. `Development Build + Autoconnect Profiler`로 최종 비교

이유:

- 에디터는 Hierarchy, Inspector, SceneView 자체 비용이 섞인다.
- 실제 차이는 빌드에서 더 명확하게 보인다.

### Unity Profiler에서 바로 봐야 할 항목

#### CPU Usage

중점 확인:

- `Instantiate`
- `Destroy`
- `GameObject.Activate`
- `BehaviourUpdate`
- `GarbageCollector`

풀링 전에는 생성/삭제 구간에서 스파이크가 보이기 쉽다.  
풀링 후에는 그 스파이크가 줄고 `SetActive`, 재초기화 코드 쪽으로 비용이 이동한다.

#### Memory

중점 확인:

- `GC Used Memory`
- `GC Reserved Memory`
- 전체 오브젝트 수
- 테스트 시작 후 메모리 증가 패턴

풀링 후 기대 변화:

- 시작 메모리는 약간 더 높을 수 있음
- 플레이 중 증가/감소 폭은 더 안정적임

#### Timeline View

한눈에 보기 가장 좋은 화면이다.

확인 포인트:

- 특정 프레임에서 큰 스파이크가 사라졌는지
- 운석이 많이 나오는 구간에서도 프레임 타임이 더 평평해졌는지

### 가장 빠른 A/B 비교 방법

Unity에서 가장 실용적인 방법은 아래 조합이다.

1. 기존 방식 브랜치에서 Profiler 캡처 저장
2. 풀링 적용 브랜치에서 같은 조건으로 Profiler 캡처 저장
3. `Profile Analyzer` 패키지로 두 캡처를 비교

장점:

- 평균 프레임 시간
- 최악 프레임 시간
- 함수별 비용 변화

를 한 번에 비교할 수 있다.

### 메모리 쪽을 더 정확히 보고 싶을 때

`Memory Profiler` 패키지로 스냅샷을 비교한다.

추천 스냅샷 시점:

1. 게임 시작 직후
2. 30초 플레이 후
3. 게임오버 직후

이렇게 보면 아래를 구분할 수 있다.

- 풀링 때문에 "미리 잡아 둔 메모리"
- 실제 누적되고 있는 오브젝트
- 예상보다 많이 남아 있는 비활성 객체

### 눈에 바로 띄게 확인하는 가장 쉬운 방법

Profiler만 켜 놓으면 숫자는 정확하지만 즉시 체감은 어려울 수 있다.  
그래서 아래 디버그 HUD를 같이 두는 것이 좋다.

화면에 표시하면 좋은 값:

- `Active Meteors`
- `Inactive Meteors`
- `Total Pool Size`
- `Pool Expansion Count`
- `Frame Time (ms)`
- `FPS`
- `GC.Alloc/frame`

이 HUD를 두면 플레이 중에도 바로 확인할 수 있다.

### 테스트에서 실제 풀 사용량 확인하는 법

오브젝트 풀 테스트에서는 단순 FPS보다 "풀을 얼마나 자주 다 쓰는지"를 보는 것이 중요하다.

가장 빠른 확인 방법은 인게임 디버그 HUD와 Profiler를 같이 보는 것이다.

#### 필수로 기록할 값

- `Active Meteors`: 현재 활성 상태 운석 수
- `Inactive Meteors`: 현재 풀에서 대기 중인 운석 수
- `Total Pool Size`: 전체 풀 크기
- `Peak Active Meteors`: 테스트 중 최대 활성 운석 수
- `Pool Miss Count`: 풀 부족 발생 횟수
- `Pool Expansion Count`: 런타임 확장 횟수

#### 해석 방법

- `Peak Active Meteors`가 `Total Pool Size`에 자주 닿는다:
  풀이 작다. 초기 풀 크기 또는 visible cap 전략을 재검토해야 한다.
- `Pool Miss Count`가 0이 아니다:
  실제 플레이에서 풀이 부족했다는 뜻이다.
- `Pool Expansion Count`가 자주 오른다:
  초기 풀 크기를 더 키워야 한다.
- `Inactive Meteors`가 대부분 0 근처다:
  현재 풀은 항상 한계 근처에서 돌고 있다.
- `Inactive Meteors`가 지나치게 많다:
  풀 크기가 과하게 큰 것이다.

#### 가장 빠른 테스트 절차

1. 스폰 속도를 높은 난이도 조건으로 고정한다.
2. 60초 동안 플레이하거나 자동 플레이로 유지한다.
3. HUD에서 `Peak Active Meteors`, `Pool Miss Count`, `Pool Expansion Count`를 기록한다.
4. 같은 테스트를 3회 이상 반복한다.
5. 가장 큰 `Peak Active Meteors` 값에 10~20% 버퍼를 더해 최종 풀 크기 후보를 정한다.

#### 추천 성공 기준

- `Pool Miss Count = 0`
- `Pool Expansion Count`가 테스트 초반 0회 또는 매우 드묾
- `Peak Active Meteors`가 `Total Pool Size`보다 충분히 낮음
- `GC.Alloc/frame`가 낮고 프레임 타임 스파이크가 줄어듦

풀링 전:

- 전체 운석 수가 계속 생성/삭제 패턴을 보임
- 프레임 타임이 순간적으로 튀는 구간이 보임

풀링 후:

- 총 풀 크기는 일정하거나 드물게만 증가
- 활성/비활성 숫자만 변함
- 프레임 타임이 더 안정적으로 보임

### 가장 추천하는 실전 측정 순서

1. 스트레스 테스트 설정을 만든다.
2. 기존 버전에서 60초 캡처를 3회 저장한다.
3. 풀링 버전에서 같은 조건으로 60초 캡처를 3회 저장한다.
4. `Profile Analyzer`로 평균/최악 프레임을 비교한다.
5. `Memory Profiler`로 시작 직후와 60초 시점 스냅샷을 비교한다.
6. HUD로 `active / inactive / total pool / expansions`를 즉시 확인한다.

### 성공 기준 예시

풀링 적용이 잘 된 것으로 볼 수 있는 신호:

- 플레이 중 `GC.Alloc/frame`가 거의 0에 가까워짐
- `Instantiate` / `Destroy` 비용이 눈에 띄게 감소
- 프레임 타임의 최고값(max spike)이 낮아짐
- 운석이 몰리는 구간에서도 프레임 흔들림이 줄어듦

반대로 아래라면 추가 개선이 필요하다.

- 풀 확장이 너무 자주 발생함
- 재사용 시 상태 초기화 버그가 생김
- 시작 메모리가 지나치게 큼
- 풀 크기를 작게 잡아 런타임 생성이 계속 발생함

## 4. 현재 프로젝트 적용 방식

### 오브젝트 풀링 사용 방식

이 프로젝트는 운석을 `Instantiate / Destroy`로 반복 생성하는 대신, 시작 시 일정 수량을 미리 만들어 두고 필요할 때 꺼내 쓰는 Object Pool 구조를 사용한다.  
기본 흐름은 `초기 풀 생성 -> 비활성 오브젝트 재사용 -> 부족 시 확장 -> 테스트 후 최종 고정 크기 결정` 순서다.

실제 적용 의도는 아래와 같다.

- 플레이 도중 반복적인 생성/삭제 비용을 줄여 프레임 타임을 안정화한다.
- `GC.Alloc`와 가비지 컬렉션 스파이크를 줄여 모바일 환경에서 끊김을 완화한다.
- 테스트 중에는 풀 부족 여부를 관측하고, 배포 전에는 관측값을 바탕으로 적절한 고정 크기를 결정한다.

### CSV 로그 추출 방식

이 프로젝트의 프로파일링 모드는 단순 콘솔 로그가 아니라, 실제 플레이 중 관측한 값을 CSV 파일로 자동 저장하도록 구성했다.  
이 방식은 "풀 크기를 왜 이 수치로 정했는가"를 정량적으로 설명하기 좋아서 포트폴리오 문서화에도 유리하다.

`Is Profiling`을 켜면 아래 정보가 CSV로 기록된다.

- 1초마다 현재 `total_pool_size`, `active_count`, `visible_count`, `inactive_count`
- 풀 부족 발생 시 `POOL_EMPTY`
- 실제 풀 확장 발생 시 `POOL_EXPAND`
- 확장 후에도 사용 가능한 오브젝트가 없으면 `SPAWN_SKIPPED`
- 플레이어 충돌을 무시한 경우 `PLAYER_HIT_IGNORED`
- 한 세션 종료 시 평균 활성 운석 수, 평균 화면 노출 운석 수, 최대 화면 노출 운석 수
- 10회 단위 세션 누적 시 최근 10회 평균 요약값

이렇게 저장한 CSV는 아래 목적에 바로 사용할 수 있다.

- 적절한 초기 풀 크기와 확장 단위를 근거 있게 조정
- visible cap 사용 여부 판단
- 포트폴리오에서 "수치 기반으로 최적화했다"는 근거 자료 제시
- 테스트 결과를 스크린샷이나 표로 재정리할 때 원본 데이터로 활용

### 포트폴리오용 설명 예시

포트폴리오에는 아래처럼 정리하면 전달력이 좋다.

- 운석 시스템을 `Instantiate / Destroy` 구조에서 Object Pool 구조로 전환했다.
- 플레이 중 1초 단위로 풀 사용량과 화면 노출 운석 수를 CSV로 기록했다.
- 풀 부족, 풀 확장, 스폰 생략 같은 이벤트를 함께 수집해 병목 구간을 확인했다.
- 10회 단위 평균 데이터를 기준으로 초기 풀 크기와 확장 단위를 조정했다.
- 결과적으로 메모리 사용 패턴을 안정화하고, 반복 생성/삭제에 따른 GC 부담을 줄이는 방향으로 구조를 개선했다.

### 인스펙터 설정 항목 설명

#### Object Pool

- `Initial Pool Size`: 시작할 때 미리 만들어 둘 운석 오브젝트 개수
- `Pool Expansion Size`: 풀이 부족할 때 한 번에 추가 생성할 운석 개수
- `Allow Pool Expansion`: 풀이 비었을 때 자동으로 풀 크기를 늘릴지 여부

#### Profiling

- `Is Profiling`: 프로파일링 모드 활성화 여부, 켜면 플레이어가 운석에 맞아도 죽지 않고 CSV 로그를 기록

#### Spawn Timing

- `Starting Spawn Interval`: 게임 시작 직후 운석이 생성되는 기본 간격
- `Minimum Spawn Interval`: 난이도가 올라간 뒤 줄어들 수 있는 최소 생성 간격
- `Spawn Ramp Duration`: 시작 간격에서 최소 간격까지 도달하는 데 걸리는 시간

#### Visible Meteor Cap

- `Use Visible Meteor Cap`: 동시에 활성화 가능한 운석 수 제한을 사용할지 여부
- `Starting Max Visible Meteors`: 게임 초반에 허용할 최대 활성 운석 수
- `Maximum Visible Meteors`: 시간이 지나도 넘지 않게 할 최대 활성 운석 수
- `Visible Meteor Increase Interval`: 활성 운석 상한을 1씩 늘리는 시간 간격

#### Meteor Speed

- `Base Meteor Speed`: 운석의 기본 이동 속도
- `Maximum Speed Variation`: 기본 속도에서 랜덤으로 더해지거나 빠질 수 있는 최대 폭
- `Speed Variation Unlock Time`: 속도 랜덤 변화가 적용되기 시작하는 시간
- `Speed Variation Ramp Duration`: 속도 랜덤 변화 폭이 최대치까지 커지는 데 걸리는 시간
- `Planet Hit Bonus Score`: 운석이 행성에 맞았을 때 추가할 점수

#### Trajectory Mix

- `Random Trajectory Chance`: 행성 정중앙 대신 랜덤 목표 지점으로 날아갈 확률
- `Random Target Padding`: 랜덤 목표 지점을 카메라 화면 밖으로 얼마나 더 넓게 잡을지

#### Spawn Placement

- `Spawn Padding`: 카메라 화면 바깥에서 얼마나 떨어진 위치에 운석을 생성할지
